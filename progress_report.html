Changes Made This Session

1. Admin-Only Course Management

Courses can no longer be created by regular users. Only admins can add/edit/delete courses. Users select from the admin-managed list when creating a listing.

new supabase/migrations/20260410_admin_only_course_management.sql

create policy "courses_insert_admin_only" on public.courses for insert to authenticated
with check (exists (select 1 from public.profiles where id = auth.uid() and role = 'admin'));

create policy "courses_update_admin_only" on public.courses for update to authenticated
using (exists (select 1 from public.profiles where id = auth.uid() and role = 'admin'));

create policy "courses_delete_admin_only" on public.courses for delete to authenticated
using (exists (select 1 from public.profiles where id = auth.uid() and role = 'admin'));

modified app/api/listings/route.ts — removed course auto-creation logic, removed "course" from FieldName type.

modified components/create-listing-form.tsx — removed free-text course input, always shows dropdown. If no courses exist: shows "No courses have been added yet. An admin will add courses soon."

2. Username + Profile Auto-Creation on Sign-Up

Sign-up form now collects first name, last name, and username. A database trigger auto-creates the profile row on registration.

new supabase/migrations/20260410_create_profile_on_signup.sql

create or replace function public.handle_new_user()
returns trigger as $$
begin
  insert into public.profiles (id, email, username, first_name, last_name)
  values (
    new.id, new.email,
    new.raw_user_meta_data->>'username',
    coalesce(new.raw_user_meta_data->>'first_name', ''),
    coalesce(new.raw_user_meta_data->>'last_name', '')
  );
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute procedure public.handle_new_user();

modified components/sign-up-form.tsx — added first name, last name, username fields. Username auto-lowercased. All passed as options.data to Supabase signUp.

3. Edit Listing

Users can edit their own listings. Admins can edit any listing. Edit link appears on listing cards only for the owner or an admin.

new supabase/migrations/20260411_listings_update_own.sql

create policy "listings_update_own" on public.listings for update to authenticated
using (
  seller_id = auth.uid()
  or exists (select 1 from public.profiles where id = auth.uid() and role = 'admin')
);

new app/api/listings/[id]/route.ts — PATCH handler. Checks user is owner or admin, validates fields, updates listing.

new app/(app)/listings/[id]/edit/page.tsx — server page, fetches listing, checks ownership, renders edit form.

new components/edit-listing-form.tsx — pre-filled form, submits PATCH, redirects to /home on success.

modified lib/marketplace.ts — added getListingById(), added "role" to ProfileSummary type and select query.

modified components/listing-card.tsx — added viewerId and viewerRole props, shows Edit link for owner/admin.

modified app/(app)/home/page.tsx — passes profile?.id and profile?.role to each ListingCard.
